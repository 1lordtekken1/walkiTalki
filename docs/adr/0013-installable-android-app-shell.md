# ADR 0013: Installable Android App Shell

## Status

Accepted.

## Context

The framework-free `:core` and browser `:preview` slices proved the protocol, diagnostics, permission policy, PTT state, and rendering copy without producing an installable Android artifact. Review feedback required an APK build and screenshots of the running app surface, while repository rules still require Android framework calls to stay outside `:core` and outside direct Bluetooth/audio production code until those adapters have dedicated seams and tests.

## Decision

Add a minimal `:app` module that builds a debug APK and hosts a single `MainActivity` with a custom Android `View` for the current PTT shell. The app depends on `:core`, but its first UI slice only renders static, privacy-safe status copy: no Bluetooth socket, `AudioRecord`, `AudioTrack`, raw peer identifier, MAC address, or audio payload is read or logged.

The screen copy is isolated in `AppUiCopy`, the MVP demo flow is coordinated by `AppTalkController`, and scan/scanning/ready/transmitting plus support-diagnostics behavior is covered by JVM unit tests before the Android `Activity` and `View` implementation. Reproducible screenshots live under `docs/screenshots/` and are generated by the root `renderAppScreenshots` task from `AppScreenshotCatalog`, so visual review uses the same `AppTalkController` and `TalkScreenPresenter` state even when headless CI has no device.

## Acceptance Criteria

- `:app:testDebugUnitTest` validates the Android app shell copy, scan-to-ready MVP demo flow, press/release PTT state transition, and privacy-safe support report before app production code is accepted.
- `:app:assembleDebug` produces `app/build/outputs/apk/debug/app-debug.apk`.
- `packageMvpDebugApk` copies the manually testable APK to `artifacts/apk/walkitalki-mvp-debug.apk` with a SHA-256 checksum.
- `renderAppScreenshots` regenerates the committed ready and active PTT screenshots from app/controller state and writes a screenshot README with diagnostics signals.
- `./gradlew test` still runs the core, preview, and app JVM checks.

## Rollback / Pivot Trigger

Rollback the `:app` module if Android Gradle Plugin/SDK setup blocks deterministic CI builds or if the app starts bypassing core seams to call Bluetooth/audio APIs directly from UI code. Pivot to Compose only after ViewModel/state tests exist for the same screen contract.

## Diagnostics / Privacy Signal

The app shell displays only redacted diagnostics guidance and can render a support report containing coarse UI diagnostics signals only. Any future transport/audio adapter must emit diagnostics through `DiagnosticsSink` and must not log raw audio payloads, raw peer identifiers, MAC addresses, or unredacted device names.
